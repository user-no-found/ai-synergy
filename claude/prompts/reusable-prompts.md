# 可复用提示词库

AI协同方案内部提示词库。

## 1. 八荣八耻原则

**用途**: 全局行为准则，适用于所有 skills 和子代理

```markdown
### Claude Code 八荣八耻

- 以瞎猜接口为耻，以认真查询为荣
- 以模糊执行为耻，以寻求确认为荣
- 以臆想业务为耻，以人类确认为荣
- 以创造接口为耻，以复用现有为荣
- 以跳过验证为耻，以主动测试为荣
- 以破坏架构为耻，以遵循规范为荣
- 以假装理解为耻，以诚实无知为荣
- 以盲目修改为耻，以谨慎重构为荣

**八条准则**:
1. 不猜接口，先查文档
2. 不糊里糊涂干活，先把边界问清
3. 不臆想业务，先跟人类对齐需求并留痕
4. 不造新接口，先复用已有
5. 不跳过验证，先写用例再跑
6. 不动架构红线，先守规范
7. 不装懂，坦白不会
8. 不盲改，谨慎重构
```

**已应用到**: `ai-agent/references/eight-principles.md`

## 2. 胶水开发原则

**用途**: 编程子代理开发时遵循，避免重复造轮子

```markdown
### 胶水开发要求

**总体原则**:
尽可能减少自行实现的底层与通用逻辑，优先、直接、完整地复用既有成熟仓库与库代码，仅在必要时编写最小业务层与调度代码。

**依赖使用规范**:
- 实际加载与执行的必须是完整、生产级实现，而非简化版本
- 禁止复制代码到当前项目后再修改使用
- 禁止对依赖模块进行功能裁剪、逻辑重写或降级封装

**功能完整性约束**:
- 所有被调用的能力必须来自依赖库的真实实现
- 不允许 Mock / Stub / Demo 替代
- 若依赖库已提供功能，禁止自行重写同类逻辑

**当前项目职责边界**:
- 允许：业务流程编排、模块组合与调度、参数配置、输入输出适配
- 禁止：重复实现算法、重写已有数据结构、拆出依赖库逻辑自己写

**评价标准**:
不是"写了多少代码"，而是"是否正确、完整地站在成熟系统之上构建新系统"。
```

**建议应用到**: `rust-agent`, `python-agent`, `c-agent` 等编程子代理

## 3. 流程标准化模板

**用途**: 将任意内容转化为标准化流程文档

```markdown
### 流程标准化输出要求

输出格式必须使用 Markdown 数字序号语法，整体表达必须直接、精准、详细。

**文档结构**（必须且只能包含）:
1. 目的
2. 适用范围
3. 注意事项
4. 相关模板或工具（如适用）
5. 流程步骤（使用 Markdown 数字编号 1, 2, 3 …）

**要求**:
- 使用简明、直接、易懂的语言
- 步骤必须可执行、按时间顺序排列
- 每一步都要明确详细具体怎么做
- 如果输入内容不完整，智能补全合理的默认流程
```

**建议应用到**: `doc-agent` 生成文档时

## 4. 项目上下文文档生成

**用途**: 生成完整的项目上下文文档，用于跨会话复用

```markdown
### 项目上下文文档结构

**核心模块**:
1. 项目概要（名称、背景、目标、要解决的问题、整体愿景）
2. 范围定义（当前范围、非本次范围、约束条件）
3. 关键实体与关系（核心实体、实体职责、实体关系描述）
4. 功能模块拆解（模块列表、模块详情、典型用户场景）
5. 技术方向与关键决策（客户端、服务端、模型层、数据流架构）
6. 交互与风格约定（AI输出风格、表达规范、格式要求）
7. 当前进展总结（已确认事实、未解决问题）
8. 后续计划与风险（待讨论主题、潜在风险、推荐后续Prompt）

**重要规则**:
- 若某字段未明确出现或无法推断，必须保留字段并填写"暂无信息"
- 不得自行虚构事实，不得省略字段
- 输出内容必须结构稳定、层级清晰、可直接复制使用
```

**建议应用到**: `project-bootstrap` skill 生成项目初始化文档

## 使用方式

1. **直接引用**: 在子代理的 `references/` 中创建对应文件
2. **嵌入硬性规则**: 将关键原则嵌入子代理的 `### 硬性规则`
3. **作为模板**: 在生成文档时参考这些结构
